Create a React context provider that maintains a global SSE connection to `/api/v1/renders/events` and triggers toast notifications on render completion/failure. Per locked decisions:
- "Toast notification with subtle completion sound when a render completes — visible on any page within the app"
- "Failed renders use same toast pattern with red styling: 'Render failed — View details'"

**'use client' component.**

**Context interface:**
```typescript
interface RenderEventContextValue {
  isConnected: boolean;
  activeCount: number;
}

const RenderEventContext = createContext<RenderEventContextValue>({
  isConnected: false,
  activeCount: 0,
});

export const useRenderEventContext = () => useContext(RenderEventContext);
```

**Component implementation:**

Props: `{ children: React.ReactNode }`

State: `isConnected`, `activeCount`

**SSE Connection (useEffect):**
- Create `new EventSource('/api/v1/renders/events')`
- Handle events same as the hook in use-render-events.ts, but this provider focuses on side effects (toasts, sound) rather than data management.
- On `connected` event: set isConnected=true, set activeCount from event data.
- On `progress` event: Track active render IDs in a Set (useRef). When a new renderId appears, increment activeCount. Maintain the set for accurate counting.
- On `completed` event: Decrement activeCount. Remove renderId from active set. Show success toast. Play sound.
- On `failed` event: Decrement activeCount. Remove renderId from active set. Show failure toast.
- Implement exponential backoff reconnection (same pattern as use-render-events.ts).
- Cleanup: close EventSource, clear timeouts.

IMPORTANT: This provider creates its OWN EventSource connection, separate from the one in `useRenderEvents` hook. This is intentional — the provider runs globally (in layout), while the hook runs only on the renders page. Two SSE connections per tab is well within the browser limit of 6. This avoids complex shared state between a context provider and a page-specific hook.

**Toast notifications (per locked decision — "visible on any page within the app"):**

On `completed` event:

**Completion sound (per locked decision: "subtle completion sound"):**

Generate a short, subtle "ding" sound programmatically using the Web Audio API. This avoids needing audio files and licensing concerns.

```typescript
function playCompletionSound() {
  try {
    const audioContext = new (window.AudioContext || (window as any).webkitAudioContext)();
    const oscillator = audioContext.createOscillator();
    const gainNode = audioContext.createGain();

    oscillator.connect(gainNode);
    gainNode.connect(audioContext.destination);

    // Subtle high-pitched ding
    oscillator.frequency.setValueAtTime(880, audioContext.currentTime); // A5 note
    oscillator.type = 'sine';

    // Quick fade in and out (0.15 seconds total)
    gainNode.gain.setValueAtTime(0, audioContext.currentTime);
    gainNode.gain.linearRampToValueAtTime(0.15, audioContext.currentTime + 0.01); // Quiet
    gainNode.gain.linearRampToValueAtTime(0, audioContext.currentTime + 0.15);

    oscillator.start(audioContext.currentTime);
    oscillator.stop(audioContext.currentTime + 0.15);
  } catch {
    // Audio not available (autoplay policy, no audio device, etc.) — ignore silently
  }
}
```

This produces a very subtle, short sine wave tone at A5 (880Hz). The gain is kept low (0.15) so it's noticeable but not startling. The try/catch handles browsers that block audio (autoplay policy requires prior user interaction). The sound is ~150ms long.

**Toast deduplication:** To prevent duplicate toasts if somehow the same completion event fires twice, use Sonner's built-in ID mechanism:

```typescript
});
```
