<objective>
Integrate credit enforcement into the render submission pipeline so every render deducts credits atomically and blocks when credits are exhausted or payments have failed.

Purpose: This is the revenue gate. Without credit enforcement, users render unlimited videos for free. With it, every render has a cost, exhaustion returns clear 402 errors, and system failures get automatic refunds.
Output: Updated render and batch APIs with credit checks, credit utility functions, and auto-refund in the render worker.
</objective>

**Create `editor/src/lib/credits.ts`** with the following exported functions:

**1. `deductCreditsForRender(organizationId: string, creditsRequired: number): Promise<DeductResult>`**
Where `DeductResult = { success: true; newBalance: number } | { success: false; available: number; required: number }`.

Implementation:
- Use `prisma.$transaction()` with `{ isolationLevel: 'RepeatableRead', timeout: 10000 }`
- Inside transaction:
  - `findUniqueOrThrow` Organization, select creditBalance
  - If creditBalance < creditsRequired, return `{ success: false, available: creditBalance, required: creditsRequired }`
  - Calculate balanceBefore, balanceAfter
  - Update Organization creditBalance to balanceAfter
  - Create CreditTransaction with amount: -creditsRequired, reason: 'render', metadata: { creditsRequired }
  - Return `{ success: true, newBalance: balanceAfter }`
- IMPORTANT: The return inside the transaction callback must be the final result. Catch the INSUFFICIENT_CREDITS case by returning the failure result from within the transaction (don't throw for insufficient credits -- only throw for actual database errors).

**2. `refundCredits(organizationId: string, creditsToRefund: number, renderId: string): Promise<void>`**
- Use Prisma transaction with RepeatableRead
- Read current balance, add creditsToRefund
- Update Organization creditBalance
- Create CreditTransaction: amount: +creditsToRefund, reason: 'refund_system_failure', renderId, metadata: { renderId }
- Reset lowCreditWarningShown if balance now above threshold (use isLowCredit from billing.ts)

**3. `checkAndWarnLowCredits(organizationId: string): Promise<{ shouldShowBanner: boolean }>`**
- Read Organization: creditBalance, monthlyAllotment, lowCreditWarningShown, lowCreditEmailSentAt, tier
- If tier is 'free' and monthlyAllotment is 0, skip (no allotment to compare against)
- Use `isLowCredit(creditBalance, monthlyAllotment)` from billing.ts
- If low AND not already warned:
  - Update Organization lowCreditWarningShown = true
  - Send email via Resend (optional pattern: if RESEND_API_KEY configured):
    - Find org owner email (Member where role='owner', join User)
    - Subject: "Low Credit Warning - OpenVideo"
    - Body: Current balance, monthly allotment, percentage, link to /dashboard/billing
    - Only send if lowCreditEmailSentAt is null or older than 24 hours
    - Update lowCreditEmailSentAt
  - Return `{ shouldShowBanner: true }`
- Return `{ shouldShowBanner: false }`

**Changes to `editor/src/workers/render-worker.ts`:**

1. Add import at top:
   ```typescript
   import { refundCredits } from '@/lib/credits';
   import { calculateCredits } from '@/lib/billing';
   ```

2. In the catch block (after `formatRenderError` and DB update), add credit refund logic:
   ```typescript
   // Auto-refund credits for system failures (not user errors)
   // Per research: refund for INTERNAL_ERROR and RENDER_TIMEOUT, not VALIDATION_ERROR or RESOURCE_MISSING
   if (category === 'INTERNAL_ERROR' || category === 'RENDER_TIMEOUT') {
     try {
       // Calculate how many credits were deducted for this render
       // Look up the CreditTransaction for this render
       const creditTx = await prisma.creditTransaction.findFirst({
         where: { renderId: job.data.renderId, reason: 'render' },
         select: { amount: true }
       });

       if (creditTx && creditTx.amount < 0) {
         const refundAmount = Math.abs(creditTx.amount);
         await refundCredits(job.data.organizationId, refundAmount, job.data.renderId);
         console.log(`[Worker] Refunded ${refundAmount} credits for system failure on render ${job.data.renderId}`);
       }
     } catch (refundError) {
       // Log but don't fail - credit refund is best-effort
       console.error(`[Worker] Failed to refund credits for render ${job.data.renderId}:`, refundError);
     }
   }
   ```

3. Place this AFTER the webhook enqueue call in the catch block (so webhooks fire regardless of refund outcome).

4. Also add the renderId to the CreditTransaction created during deduction. In `lib/credits.ts` `deductCreditsForRender`, add an optional `renderId` parameter that gets passed to the CreditTransaction.create call. Update the render route.ts to pass `render.id` after creating the Render record. Actually, since credits are deducted BEFORE the render is created, we need to update the transaction after render creation. Alternative approach: pass renderId as null during deduction, then the worker can still find the transaction by looking at the most recent render-reason transaction for this org. OR -- better: add a `referenceId` to the deduction that matches a unique identifier. The simplest approach: in the render route, after creating the Render record, update the CreditTransaction to add the renderId:
   ```typescript
   // After render creation, link the credit transaction
   await prisma.creditTransaction.updateMany({
     where: {
       organizationId: context.organizationId,
       reason: 'render',
       renderId: null,
       createdAt: { gte: new Date(Date.now() - 5000) } // Within last 5 seconds
     },
     data: { renderId: render.id },
     // Take only the most recent one
   });
   ```
   Actually this is fragile. Better approach: Create the CreditTransaction with a temporary marker, then update after render creation. Simplest: just store credits deducted in the job data so the worker knows how much to refund without looking up the transaction:

   In renders/route.ts, add `creditsDeducted: creditsRequired` to the job data passed to `renderQueue.add()`.
   In batch/route.ts, add `creditsDeducted: perRenderCredits` to each job's data.
   In the worker, read `job.data.creditsDeducted` for the refund amount.

   This is cleaner and avoids the lookup problem entirely. Update the deduction to also set renderId: null in the CreditTransaction, and have the worker set it after the fact if needed for audit.

So the final approach:
- In renders/route.ts: pass `creditsDeducted` in job data
- In batch/route.ts: pass `creditsDeducted: perRenderCredits` in each job
- In worker: use `job.data.creditsDeducted` for refund amount on system failure
- This avoids needing to look up CreditTransaction by renderId
