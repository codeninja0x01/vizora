  <name>Task 1: Stripe webhook route with signature verification and idempotency</name>
  <files>editor/src/app/api/webhooks/stripe/route.ts</files>
  <action>
Create `editor/src/app/api/webhooks/stripe/route.ts` as a Next.js App Router route handler.

**CRITICAL:** The webhook route MUST read the raw request body as text (not JSON) for Stripe signature verification. In Next.js App Router, use `await request.text()` to get the raw body before `stripe.webhooks.constructEvent()`.

**Implementation:**

1. Export a `POST` async function that:
   - Reads raw body via `request.text()`
   - Gets `stripe-signature` header from the request
   - Returns 400 if STRIPE_WEBHOOK_SECRET is not configured
   - Calls `stripe.webhooks.constructEvent(body, signature, webhookSecret)` to verify signature
   - On verification failure, return 400 with "Webhook signature invalid"

2. After verification, perform idempotency check:
   - Look up `prisma.subscriptionEvent.findUnique({ where: { stripeEventId: event.id } })`
   - If found AND processed=true, return 200 "Event already processed"

3. Record the event immediately (before processing):
   - Upsert into SubscriptionEvent with `stripeEventId`, `eventType: event.type`, `processed: false`
   - For organizationId: extract from event metadata or customer lookup (see below)

4. Route to handler based on `event.type`:
   - `checkout.session.completed` -> handleCheckoutComplete
   - `invoice.paid` -> handleInvoicePaid
   - `invoice.payment_failed` -> handlePaymentFailed
   - `customer.subscription.updated` -> handleSubscriptionUpdated
   - `customer.subscription.deleted` -> handleSubscriptionDeleted
   - Default: log unhandled event type, still return 200

5. After handler succeeds, mark event as processed:
   - `prisma.subscriptionEvent.update({ where: { stripeEventId: event.id }, data: { processed: true } })`

6. Return 200 "OK"

7. Wrap entire handler in try-catch. On unexpected errors, return 500 but log the error.

**Helper function** `getOrganizationIdFromEvent(event: Stripe.Event): Promise<string | null>`:
- For checkout.session.completed: read `event.data.object.metadata.organizationId`
- For invoice events: look up organization by `stripeCustomerId` matching `event.data.object.customer`
- For subscription events: look up organization by `stripeCustomerId` matching `event.data.object.customer`
- Return null if organization not found (log warning, return 200 to acknowledge)

Import stripe from `@/lib/stripe`, prisma from `@/lib/db`, handlers from `@/lib/stripe-handlers`.
If stripe is null (not configured), return 503 "Billing not configured".

<task type="auto">
  <name>Task 2: Subscription lifecycle event handlers</name>
  <files>editor/src/lib/stripe-handlers.ts</files>
  <action>
Create `editor/src/lib/stripe-handlers.ts` with handler functions for each Stripe event type.

Import Stripe types, prisma from `@/lib/db`, TIER_CONFIG from `@/lib/billing`, Resend pattern from `@/lib/auth` (use same optional Resend pattern: check RESEND_API_KEY, log if missing).

**1. handleCheckoutComplete(event: Stripe.Event)**
- Extract session as `Stripe.Checkout.Session` from `event.data.object`
- Get organizationId from `session.metadata.organizationId`
- If `session.mode === 'subscription'`:
  - Retrieve subscription via `stripe.subscriptions.retrieve(session.subscription as string)`
  - Determine tier from subscription price ID: match against `process.env.STRIPE_PRO_PRICE_ID` -> 'pro'. Default to 'pro' if no match (enterprise is manual).
  - Update Organization: set `stripeCustomerId` (if not already set), `stripeSubscriptionId`, `tier`, `subscriptionStatus: 'active'`, `monthlyAllotment` from TIER_CONFIG, `creditBalance` += monthlyAllotment (initial credit grant), `billingCycleStart: new Date()`, `billingCycleEnd` (1 month from now using date-fns `addMonths`), `lastCreditReset: new Date()`
  - Create CreditTransaction for initial credit grant (reason: 'subscription_renewal')
- If `session.mode === 'payment'` (credit pack):
  - Get packSize from `session.metadata.packSize` (number)
  - Use a Prisma transaction with RepeatableRead isolation:
    - Read Organization.creditBalance
    - Add packSize to balance
    - Update Organization.creditBalance
    - Create CreditTransaction with reason: 'credit_pack', metadata: { packSize }
  - Reset lowCreditWarningShown if balance now above threshold

**2. handleInvoicePaid(event: Stripe.Event)**
- Extract invoice as `Stripe.Invoice` from `event.data.object`
- Skip if `invoice.billing_reason === 'subscription_create'` (handled by checkout.session.completed)
- Find organization by stripeCustomerId matching `invoice.customer`
- If not found, log and return
- Perform credit rollover in a Prisma transaction:
  - Read current creditBalance, monthlyAllotment, tier
  - Calculate new balance: `Math.min(currentBalance + monthlyAllotment, monthlyAllotment * 2)` (2x cap per user decision)
  - Calculate added credits: `newBalance - currentBalance`
  - Update Organization: `creditBalance: newBalance`, `billingCycleStart: new Date()`, `billingCycleEnd: addMonths(new Date(), 1)`, `lastCreditReset: new Date()`, `lowCreditWarningShown: false`, `lowCreditEmailSentAt: null`
  - Create CreditTransaction: reason 'subscription_renewal', amount: addedCredits, metadata with rollover details

**3. handlePaymentFailed(event: Stripe.Event)**
- Extract invoice as `Stripe.Invoice` from `event.data.object`
- Find organization by stripeCustomerId
- **Immediately suspend:** Update Organization `subscriptionStatus: 'past_due'` (no grace period per user decision)
- Send email notification via Resend (use same optional pattern):
  - Subject: "Payment Failed - Rendering Suspended"
  - Body: Inform user their payment failed, rendering is suspended, include link to `/dashboard/billing` to update payment method
  - To: Find organization owner's email (query Member where role='owner', join User for email)
- Log the suspension

**4. handleSubscriptionUpdated(event: Stripe.Event)**
- Extract subscription as `Stripe.Subscription` from `event.data.object`
- Find organization by stripeCustomerId
- Sync fields:
  - `subscriptionStatus: subscription.status`
  - `cancelAtPeriodEnd: subscription.cancel_at_period_end`
  - If price changed (tier upgrade/downgrade): update `tier` and `monthlyAllotment` from TIER_CONFIG
  - Do NOT change creditBalance on tier change mid-cycle (per research recommendation: preserve balance, adjust allotment on next cycle)

**5. handleSubscriptionDeleted(event: Stripe.Event)**
- Find organization by stripeCustomerId
- Update Organization: `tier: 'free'`, `subscriptionStatus: 'canceled'`, `stripeSubscriptionId: null`, `monthlyAllotment: TIER_CONFIG.free.monthlyAllotment`, `cancelAtPeriodEnd: false`
- Do NOT zero out creditBalance (they keep remaining credits until next cycle)
- Create CreditTransaction if monthlyAllotment changed, recording the transition

Use `date-fns` (already installed) for `addMonths`. All database updates that touch creditBalance MUST use Prisma transactions with `isolationLevel: 'RepeatableRead'` to prevent race conditions.
