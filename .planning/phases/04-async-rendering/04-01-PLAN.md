  <action>
Install BullMQ and ioredis in the editor package:
```bash
cd editor && pnpm add bullmq ioredis
```

**Create `editor/src/lib/redis.ts`** - ioredis connection singleton:
- Follow the globalThis singleton pattern from db.ts (survives HMR in development)
- Use `new Redis()` from ioredis with config from env vars:
  - `REDIS_HOST` (default: '127.0.0.1')
  - `REDIS_PORT` (default: '6379')
  - `REDIS_PASSWORD` (optional)
- CRITICAL BullMQ requirements: set `maxRetriesPerRequest: null` and `enableReadyCheck: false` (BullMQ will error without these)
- Add error and connect event handlers with `[Redis]` prefix logging
- Export `redisConnection`

**Create `editor/src/lib/queue.ts`** - BullMQ render queue:
- Import `Queue` from 'bullmq' and `redisConnection` from './redis'
- Create and export `renderQueue = new Queue('render-queue', { connection: redisConnection, ... })`
- Default job options per user decision (no auto-retry):
  - `attempts: 1` (no auto-retry)
  - `removeOnComplete: { age: 30 * 24 * 60 * 60, count: 10000 }` (30-day retention)
  - `removeOnFail: { age: 30 * 24 * 60 * 60, count: 5000 }` (30-day retention)

**Create `editor/src/lib/error-categorization.ts`** - typed error categories:
- Export `ErrorCategory` type: `'VALIDATION_ERROR' | 'RENDER_TIMEOUT' | 'RESOURCE_MISSING' | 'INTERNAL_ERROR'`
- Export `categorizeError(error: unknown): ErrorCategory` function:
  - 'not found' / 'does not exist' -> RESOURCE_MISSING
  - 'timeout' / 'Timeout' / 'exceeded' -> RENDER_TIMEOUT
  - 'validation' / 'invalid' / 'required' -> VALIDATION_ERROR
  - Everything else -> INTERNAL_ERROR
- Export `formatRenderError(error: unknown): { category: ErrorCategory; message: string }` helper

**Update `editor/.env.sample`** - add Redis configuration:
- Add `REDIS_HOST=127.0.0.1` with comment
- Add `REDIS_PORT=6379` with comment
- Add `REDIS_PASSWORD=` with comment noting it's optional for local dev

  <action>
Add the Render model to `editor/prisma/schema.prisma` after the Template model.

**Render model fields:**
- `id` - String @id @default(cuid())
- `status` - String (values: 'queued', 'active', 'completed', 'failed')
- `templateId` - String
- `userId` - String
- `organizationId` - String
- `mergeData` - Json? (store the submitted merge data for debugging/audit)
- `outputUrl` - String? (set on completion, e.g., `/renders/{id}.mp4`)
- `errorCategory` - String? (ErrorCategory values from error-categorization.ts)
- `errorMessage` - String? (detailed error message with context)
- `queuedAt` - DateTime @default(now())
- `startedAt` - DateTime?
- `completedAt` - DateTime?
- `failedAt` - DateTime?

**Relations:**
- `user` - User @relation(fields: [userId], references: [id], onDelete: Cascade)
- `organization` - Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)
- `template` - Template @relation(fields: [templateId], references: [id], onDelete: Restrict)

**Indexes (per research - critical for API query performance):**
- `@@index([organizationId, status, queuedAt])` - primary query pattern for list endpoint
- `@@index([organizationId, templateId])` - filter by template
- `@@index([status, queuedAt])` - global status queries

**Update existing models to include reverse relations:**
- Add `renders Render[]` to User model
- Add `renders Render[]` to Organization model
- Add `renders Render[]` to Template model

After schema changes, run:
```bash
cd editor && npx prisma db push
```

Note: Use `onDelete: Restrict` for Template relation to prevent deleting templates that have render history.
