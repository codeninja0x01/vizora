Create `editor/src/app/api/v1/renders/batch/route.ts` following the pattern from the existing POST /api/v1/renders endpoint.

**Request validation with Zod schema:**
```typescript
const batchRenderSchema = z.object({
  templateId: z.string().min(1),
  mergeDataArray: z.array(z.record(z.string(), z.unknown())).min(1),
  options: z.object({
    width: z.number().int().positive().optional(),
    height: z.number().int().positive().optional(),
    fps: z.number().int().min(1).max(120).optional(),
    quality: z.number().min(0).max(1).optional(),
  }).optional(),
});
```

**POST handler implementation (wrapped in withApiAuth):**

1. Parse and validate request body against batchRenderSchema. Return 400 on invalid.

2. Check batch size against tier limits (import BATCH_SIZE_LIMITS from batch/types):
   - `if (mergeDataArray.length > BATCH_SIZE_LIMITS[context.tier]) return 400` with message showing limit
   - Per user decision: Free: 10, Pro: 100, Enterprise: 1000

3. Fetch template and verify access (same pattern as existing renders POST):
   - Template must exist (404 if not)
   - Template must be public OR owned by same organization (404 if not)

4. Eager validation of ALL merge data rows before queuing ANY renders:
   - Get mergeFields from template
   - Loop through mergeDataArray and call validateMergeData() for each row
   - Collect all validation results with rowIndex
   - If ANY row fails, return 400 with `{ error: 'Validation failed', invalidRows: [{ index, errors }] }`
   - Per user decision: preview/dry-run required on dashboard, but API validates all rows up front

5. Also check per-tier concurrent queue limits (same as existing renders POST):
   - Count active renders for org, check against tierLimits (free: 5, pro: 50, enterprise: Infinity)
   - If adding batch would exceed, return 429

6. Create Batch record in database:
   ```typescript
   const batch = await prisma.batch.create({
     data: {
       organizationId: context.organizationId,
       templateId,
       totalCount: mergeDataArray.length,
       status: 'queued',
     }
   });
   ```

7. Create all Render records in database (source of truth, before queue):
   ```typescript
   const renders = await Promise.all(
     mergeDataArray.map((mergeData, index) =>
       prisma.render.create({
         data: {
           status: 'queued',
           templateId,
           userId: context.userId,
           organizationId: context.organizationId,
           mergeData: mergeData as any,
           batchId: batch.id,
           batchIndex: index,
         }
       })
     )
   );
   ```

8. Queue all renders via queueBatchRenders (uses chunked addBulk):
   - Pass render IDs as jobIds for correlation
   - Catch queue errors: if queuing fails, update batch status to 'failed' and render statuses to 'failed'

9. Return 202 Accepted:
   ```json
   {
     "id": "batch_id",
     "status": "queued",
     "totalCount": N,
     "templateId": "...",
     "createdAt": "ISO string"
   }
   ```
   With headers: `Location: /api/v1/batches/{batchId}`, `Retry-After: 5`
  </action>

2. Fetch batch with organization scope, return 404 if not found
3. Find all failed renders for this batch:

     where: { batchId, status: 'failed' },
   });

4. Return 400 if no failed renders: `{ error: 'No failed renders to retry' }`
