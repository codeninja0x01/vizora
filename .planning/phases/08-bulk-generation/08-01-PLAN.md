// Batch System (Bulk Generation)

model Batch {
  id             String    @id @default(cuid())
  organizationId String
  templateId     String
  totalCount     Int
  status         String    @default("queued") // queued, processing, completed, partial_failure, failed
  createdAt      DateTime  @default(now())

  // Relations
  organization Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  template     Template     @relation(fields: [templateId], references: [id], onDelete: Restrict)
  renders      Render[]

  @@index([organizationId, status])
  @@index([organizationId, createdAt])
}

**Create `editor/src/lib/batch/types.ts`** - batch type definitions and constants:
- `BatchStatus` type: `'queued' | 'processing' | 'completed' | 'partial_failure' | 'failed'`
- `BatchProgress` interface: `{ batchId, total, queued, processing, completed, failed, percentComplete }`
- `FieldMapping` interface: `{ csvColumn: string, templateField: string, confidence: number }`
- `ValidationResult` interface: `{ rowIndex: number, valid: boolean, errors: string[], data: Record<string, unknown> }`
- Tier-based batch size limits constant: `BATCH_SIZE_LIMITS = { free: 10, pro: 100, enterprise: 1000 }` (per user decision)
- Chunk size constant: `BULK_CHUNK_SIZE = 100` (per research - prevent Redis timeout on large batches)

**Create `editor/src/lib/batch/queue.ts`** - BullMQ addBulk wrapper with chunking:
- Import `renderQueue` from `@/lib/queue`
- Export `queueBatchRenders(batchId, templateId, mergeDataArray, userId, organizationId)`:
  - Prepare job array where each job has `name: 'render-video'`, `data: { renderId, templateId, mergeData, batchId, batchIndex, userId, organizationId }`, `opts: { jobId: renderId }` (use the pre-created Render record ID)
  - Chunk the array into groups of `BULK_CHUNK_SIZE` (100)
  - Call `renderQueue.addBulk(chunk)` sequentially for each chunk (per research pitfall #2 - prevents Redis max command size errors)
  - Return array of all queued job IDs
- Helper function `chunkArray<T>(array: T[], size: number): T[][]`

**Create `editor/src/lib/batch/tracker.ts`** - batch progress aggregation:
- Import `prisma` from `@/lib/db`
- Export `getBatchProgress(batchId: string): Promise<BatchProgress>`:
  - Use `prisma.render.groupBy({ by: ['status'], where: { batchId }, _count: true })` for efficient single-query aggregation
  - Compute percentComplete as `((completed + failed) / total) * 100`
  - Return BatchProgress object
- Export `updateBatchStatus(batchId: string): Promise<void>`:
  - Call getBatchProgress to get current state
  - Determine batch status: all completed -> 'completed', all failed -> 'failed', any failed + rest done -> 'partial_failure', any active/queued -> 'processing'
  - Update Batch record with new status and completedAt if terminal
  - Use `prisma.batch.update()`

**Create `editor/src/lib/batch/field-matcher.ts`** - CSV column auto-mapping:
- Import `findBestMatch` from `string-similarity`
- Export `autoMapFields(csvHeaders: string[], templateFields: string[]): FieldMapping[]`:
  - For each CSV header, call `findBestMatch(csvHeader.toLowerCase(), normalizedTemplateFields)`
  - Normalize both sides: strip underscores, convert camelCase to spaces, lowercase
  - Only include matches with confidence > 0.4 (Dice coefficient threshold per research)
  - Prevent duplicate mappings: if a template field is already mapped with higher confidence, skip lower match
  - Return array of FieldMapping sorted by confidence descending
  </action>
